title: The FASTQ format
---


The FASTQ format stores sequencing reads from base calling: each record is exactly four lines.

## An example

A FASTQ file with four sequences can look like this:

```text
@SRR001666.1 071112_SLXA-EAS1_s_7:5:1:817:345 length=36
GGGTTGATGGTGCTAGCGATCTGAGACTGGTATC
+
IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
@SRR001666.2 071112_SLXA-EAS1_s_7:5:1:810:345 length=36
GCGTTGATGGTGCTAGCGATCTGAGACTGGTATC
+
IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
@SRR001666.3 071112_SLXA-EAS1_s_7:5:1:817:341 length=36
GGGTTGATGGTGCTAGCGATCTGAGACTGGTATC
+
IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
@SRR001666.4 071112_SLXA-EAS1_s_7:5:1:810:341 length=36
GCGTTGATGGTGCTAGCGATCTGAGACTGGTATC
+
IIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIIII
```

Each sequence is stored in four line:

- header (the name is the first string after the `@`, while anything after the first space is considered a comment or description
- sequence
- spacer
- quality (the probability of each base of being wrong, see below)

## The Phred quality score

The Phred score \(Q\) is the error probability on a log scale:

![Phred Score]({{ site.baseurl }}{% link img/phred.png %})

A score of 20 means 1 in 100 chance of error; 30 means 1 in 1,000.

The clever part: each score is encoded as a single ASCII character by adding 33 to the numeric value (Sanger/Phred+33). 
This keeps every base to one character, making the format four lines per read and trivial to parse with line-based tools. 
A score of 40 becomes `I` (ASCII 73), which is why quality lines often look like runs of `I` and `H`.

## Counting reads

FASTQ files have four lines per read. To count lines in a compressed file:

```bash
zcat sample.fastq.gz | wc -l
```

Divide the result by four to get the number of reads. This primitive example is just a reminder:
FASTQ files are simple text files and basic Linux commands are perfectly capable of handling them.

### With SeqFu

[SeqFu](https://github.com/telatin/seqfu2) is a package of utilities for FASTA and FASTQ files. 
It handles compression, paired-end awareness, and produces tabular summaries.

To get statistics for a single file:

```bash
seqfu stats sample.fastq.gz
```


Output:

```text
File                Reads   Bases   AvgLen  minLen  maxLen  N50
sample.fastq.gz     300000  45000000  150     35      151     150
```

For a directory of files, add `--nice` (`-n`) for human-readable spacing and `--basename` (`-b`) to print filenames without paths, and
`-t` for thousand separators. You can combine the short version of the flags as shown below:

```bash
seqfu stats -ntb *.fastq.gz
```


## Quality overview with FastQC

[FastQC](https://www.bioinformatics.babraham.ac.uk/projects/fastqc/) generates a quality report from FASTQ files. Run it on a single file:

```bash
fastqc -o sample_qc/ sample.fastq.gz
```

FastQC produces an HTML report and a ZIP file for each input. The key sections to check:

- **Per base sequence quality**: Boxplot of quality scores across read positions. Watch for yellow warning or red failure zones at the 3' end.
- **Per sequence quality scores**: Histogram of mean quality per read. Most reads should cluster at the high-quality end.
- **Per base sequence content**: Should be roughly flat for random libraries. Spikes indicate adapter contamination or non-random priming.
- **Adapter Content**: Flags if known adapter sequences are present, telling you if trimming is needed.

If per-base quality drops toward the end of the reads, trim with [fastp](https://github.com/OpenGene/fastp) or similar before proceeding.
