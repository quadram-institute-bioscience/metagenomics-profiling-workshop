title: Using Deacon
---

When analysing metagenomic reads, we often want to remove reads from the **host** before studying the microbial fraction.

For example:

* human reads in gut, vaginal, respiratory or clinical metagenomes
* mouse reads in animal experiments
* plant reads in plant-associated microbiomes (e.g. our own **Coffee**)

This step is usually called **host removal** or **host depletion**.

## Why DEACON?

A common approach is to **map** reads against a host genome with tools such as Bowtie2 or Minimap2, and then discard the reads that map.

This works, but mapping is doing more work than we usually need. 
A mapper tries to find an alignment: *where in the genome* the reads can be positioned (position, orientation, mismatches, gaps, and other details).

For host removal we usually need a simpler answer:

> Does this read look enough like the host to remove it?

DEACON answers this question using **k-mers** and **minimizers**.

A k-mer is a short DNA word. A minimizer is a representative k-mer selected from a window of sequence. Instead of aligning the full read, DEACON checks how many minimizers from the read are found in a host index.

If enough minimizers match, the read is considered a match to the indexed sequence.

DEACON can then work in two modes:

* **search mode**: keep reads that match the index
* **deplete mode**: remove reads that match the index

For host removal, we normally use **deplete mode**.

## Install DEACON

Using conda or mamba:

```bash
mamba install -c bioconda deacon
```

Check that the command is available:

```bash
deacon --help
```

## Human host? use a prebuilt human index

For human host depletion, the easiest option is to download the prebuilt `panhuman-1` index.

```bash
deacon index fetch panhuman-1
```

This will download an index called something like:

```bash
panhuman-1.k31w15.idx
```

The name tells us that the index was built with:

* `k = 31`
* `w = 15`

These are the k-mer and minimizer window parameters used when creating the index.

## And our coffee? build your own index

You can also build an index from your own FASTA or FASTQ file.

For example, to build an index from a host genome:

```bash
deacon index build host_genome.fa.gz > host.idx
```

You can inspect the index:

```bash
deacon index info host.idx
```

This reports information such as the k-mer length, window size, index format version, and number of minimizers.


## Filter single-end reads

To remove human reads from a single FASTQ file:

```bash
deacon filter -d panhuman-1.k31w15.idx reads.fq.gz -o reads.dehosted.fq.gz
```

The important option is:

```bash
-d
```

This means **deplete**: discard reads that match the index.

Without `-d`, DEACON keeps matching reads instead.

## Filter paired-end reads

For paired-end reads:

```bash
deacon filter -d panhuman-1.k31w15.idx \
  sample_R1.fq.gz sample_R2.fq.gz \
  -o sample.no-host_R1.fq.gz \
  -O sample.no-host_R2.fq.gz
```

Here:

* `sample_R1.fq.gz` is read 1
* `sample_R2.fq.gz` is read 2
* `-o` writes filtered read 1
* `-O` writes filtered read 2

For paired reads, a match in either mate is enough for the pair to be treated as matching the index.

## Save a summary report

It is useful to save a JSON summary of the filtering step:

```bash
deacon filter -d panhuman-1.k31w15.idx \
  sample_R1.fq.gz sample_R2.fq.gz \
  -o sample.no-host_R1.fq.gz \
  -O sample.no-host_R2.fq.gz \
  -s sample.deacon.summary.json
```

The summary reports how many sequences and bases were processed, removed, and retained.

This is useful for quality control and for documenting the analysis.
 

## Adjusting the filtering threshold

By default, DEACON considers a read to match the index when it passes both:

* an absolute threshold: at least 2 shared minimizers
* a relative threshold: at least 1% of minimizers shared

These can be changed with:

* `-a`: absolute threshold
* `-r`: relative threshold

For more sensitive host removal:

```bash
deacon filter -d -a 1 -r 0 \
  panhuman-1.k31w15.idx reads.fq.gz \
  -o reads.dehosted.fq.gz
```

This is more aggressive: it removes reads with even one matching minimizer.

For more specific filtering:

```bash
deacon filter -d -r 0.1 \
  panhuman-1.k31w15.idx reads.fq.gz \
  -o reads.dehosted.fq.gz
```

This requires a higher proportion of minimizers to match before the read is removed.

## Choosing sensitivity or specificity

Host removal is a trade-off.

More sensitive settings remove more host reads, but may also remove more microbial reads.

More specific settings are more conservative, but may leave more host reads behind.

A practical rule:

* if privacy or host contamination is the main concern, use more sensitive settings
* if preserving rare microbial reads is the main concern, use more conservative settings
* always inspect the summary and, if possible, compare downstream results


## A minimal workflow

For a human-associated paired-end metagenome:

```bash
# 1. Install
mamba install -c bioconda deacon

# 2. Download the prebuilt human pangenome index
deacon index fetch panhuman-1

# 3. Remove human reads
deacon filter -d panhuman-1.k31w15.idx \
  sample_R1.fq.gz sample_R2.fq.gz \
  -o sample.dehosted_R1.fq.gz \
  -O sample.dehosted_R2.fq.gz \
  -s sample.deacon.summary.json

# 4. Inspect the summary
cat sample.deacon.summary.json
```

The filtered FASTQ files can then be used for the downstream metagenomics workflow.

## Key points

DEACON is not a read mapper. It does not try to align every read to the host genome.

Instead, it uses minimizer matching to make a fast filtering decision.

This makes it useful for host depletion, especially when working with large host references or pangenomes.

For most human metagenomics projects, start with the prebuilt `panhuman-1` index and default parameters, save the JSON summary, and only tune thresholds if the default behaviour is not appropriate for your sample type.
